“Most tools bury the answer inside editors you didn’t ask for, behind build steps you don’t need.” The readtime premise

Reading-time estimation is a solved problem. The question is why every solution drags a JS framework along for the ride.

Readtime is the answer: a textarea, an Express route, and HTMX to wire them together. That’s it.

Chapter 01

The problem with reading-time tools

You want a number. They hand you a whole editor.

01

Heavyweight editors

Full-featured editors for a number you need in under a second. Markdown preview, collaboration, plugins — none of which help you estimate read time.

02

Mandatory build steps

Webpack, Vite, Babel — configured and compiled before you can estimate 50 words. The toolchain complexity dwarfs the problem it solves.

03

Manual refresh required

Click a button after every edit just to refresh the count. Not a live tool — a calculator masquerading as one.

Chapter 02

How readtime works

One textarea. One server route. HTMX wires them together.

Step 01

You type

Text enters a plain <textarea>. HTMX listens on every keyup with a 250ms debounce — fast enough to feel instant, slow enough to skip excess requests.

Step 02

HTMX POSTs

HTMX sends a POST to /readtime with the textarea value. No JSON serialization, no fetch calls, no client-side logic written by you.

Step 03

Express computes

The Express server runs the reading-time npm package and returns a tiny HTML fragment — just the text you need to display.

Step 04

Result swapped in

HTMX swaps the fragment into #readtime-result. The page never reloads. No virtual DOM. Pure hypermedia architecture.

Chapter 03

Everything you need. Nothing you don’t.

Readtime strips the problem to its essentials, then stops.

Live updates on every keystroke

Reading time recalculates via a debounced HTMX request. You never have to click anything.

No build step

Plain HTML, HTMX from CDN, a single Express route. Clone and run immediately.

250ms smart debounce

Requests stay minimal while the UI feels instant throughout your writing session.

Server-driven logic

All calculation logic lives on the server — easy to audit, easy to swap out.

Vercel-ready

Ships with a vercel.json config so you deploy in under a minute.

HTML fragment response

The server returns only the changed fragment. Minimal bandwidth, perfect hypermedia architecture.

The Stack

Four technologies. One tiny tool.

Nothing exotic. Everything justified.

HTMX Frontend

Hypermedia library that adds AJAX to HTML with attributes. Powers the live update without a single line of custom JavaScript.

Express Backend

Minimal Node.js web framework. Handles the single POST /readtime route with five lines of code.

reading‑time Library

Battle-tested npm package. Estimates read time based on average reading speed of 200 words per minute.

Vercel Hosting

Serverless hosting platform. A single vercel.json config file makes the Express app deploy instantly.
