What it does

Two functions, full pipeline

Each Lambda function intercepts a different CloudFront event, together covering the complete request-response lifecycle.

viewer-request

URI Rewriter

Intercepts every viewer request before CloudFront checks its cache. Parses the Accept header and User-Agent to detect WebP support, then rewrites the image URI to include dimension and format variant suffixes.

  • WebP detection via Accept header
  • Dimension variant key generation
  • Query param passthrough (prefix, quality)
origin-response

On-Demand Resizer

Fires only on CloudFront 404s — when a variant doesn't exist yet. Fetches the original from S3, processes it with Sharp (resize + WebP convert), stores the result back in S3, then returns the image directly without a second roundtrip.

  • Sharp-powered resize + format convert
  • Write-through S3 variant cache
  • Fires only on cache miss (404)

Request flow

Step by step

01

Browser sends request

A browser requests /images/photo.jpg?prefix=blog&d=800x600. CloudFront triggers the viewer-request Lambda before checking its cache.

02

Viewer-request rewrites URI

The function reads the Accept header. If WebP is supported, it rewrites the key to /blog/800x600/photo.webp and strips the query params.

03

CloudFront fetches from S3

CloudFront checks its cache and S3 for the variant key. On a first request the variant doesn't exist — S3 returns 404.

04

Origin-response generates variant

The 404 triggers the origin-response Lambda. It fetches the original photo.jpg from S3, resizes to 800×600 with Sharp, converts to WebP, and puts the result back in S3.

05

Browser receives optimised image

The function returns the generated WebP directly in the response. CloudFront caches it. All subsequent requests for this variant hit the cache instantly.
